# ViralBrain.ai

`0-1 AI 内容分析 SaaS / 面向 YouTube 创作者与内容运营 / 已上线并有真实付费验证`

ViralBrain.ai 不是一个单点的视频总结工具，而是一条从趋势发现、单视频拆解、爆款对标到素材沉淀的完整研究工作流。它面向需要持续判断选题、复盘内容结构、整理参考样本的 YouTube 创作者和内容操盘手，目标是把公开视频转成可复盘、可对标、可分享的结构化判断。

## Problem

YouTube 创作者和内容运营团队真正缺的，往往不是“再看更多视频”，而是：

- 快速判断哪些内容值得继续做，哪些只是偶然起量
- 解释一条视频为什么有效，而不是只记住一个标题
- 找到可复用的结构、钩子和节奏，而不是收集一堆零散链接
- 把研究结果沉淀成团队能重复调用的素材资产，而不是每次从头开始

ViralBrain.ai 解决的就是这件事：把趋势发现、深度拆解、参考样本管理和行动建议收进同一条工作流里。

## Why AI

这类问题不能只靠普通检索或手工整理，因为用户需要的不是“找到视频”，而是“形成判断”。

- 需要把公开视频、字幕、评论和表现信号转成结构化洞察
- 需要把“趋势发现 + 单视频拆解 + 爆款对标 + 素材沉淀”串成一个连贯流程
- 需要 AI 来完成摘要、结构分析、参考样本比对和行动建议输出

如果只做人工整理，成本高、速度慢，而且难以形成可复用的研究模板；如果只做简单总结，又无法支持后续的对标和决策。AI 在这里承担的是“把原始内容转成可执行判断”的角色。

## AI Workflow

项目的核心 AI 链路是 `analysis -> benchmark -> score` 三段式工作流。

### 1. analysis

- 输入：YouTube 链接、视频 metadata、字幕、评论、封面
- 输出：结构分析、封面诊断、受众与评论洞察
- 为什么单独拆出来：先把“理解这条视频本身”这件事做好，后面才能稳定进入对标和打分

### 2. benchmark

- 输入：analysis 阶段产出的结构摘要、topic hint、爆款库样本
- 输出：最相近的参考内容、共同点、差异点、可借鉴点、要避免的问题
- 为什么单独拆出来：对标检索本质上是 RAG / similarity retrieval 问题，需要独立 trace、独立优化和独立 fallback

### 3. score

- 输入：视频 metadata、analysis 结果、benchmark 结果
- 输出：总分、维度拆分、最优先行动建议
- 为什么单独拆出来：评分与建议层是最终决策层，和内容理解、样本检索属于不同职责，拆开更利于后续演进

这套设计的重点不是“调用了几个模型”，而是让每一段都能被单独观察、单独定位失败原因、单独优化。

## Engineering Trade-offs

项目里有几处刻意做出的工程取舍：

- `Next.js + FastAPI` 拆分  
  Web 层负责用户体验、鉴权、配额、支付、报告流转；AI Service 负责分析、对标、评分执行。这样可以把产品层逻辑和模型执行链路分开，便于定位问题和后续独立扩展。

- benchmark retrieval 独立成单独阶段  
  不把检索直接塞进分析阶段，是为了让 RAG 对标链路具备单独 trace、单独 fallback 和单独调优空间。

- 使用 `RLS / 配额 / Webhook / request_id / trace`  
  这个项目不是只做 demo 页面，而是按真实产品思路补齐权限、多租户隔离、额度消耗、支付回调和请求追踪，让链路能走向可交付状态。

- 区分 `preview` 与 `production` 两种运行模式  
  Preview 允许示例数据和联调兜底，方便开发与演示；Production 要求分析、趋势、采集走真实服务链路，不再静默伪装成功。

- 允许 provider fallback，但正式模式不掩盖失败  
  当外部 provider 不可用时，可以在特定模式下降级，保证开发和验证效率；但在正式模式下，系统会明确报错，而不是假装完成真实分析。

## Validation / Proof

截至 `2026.03`，项目已有：

- `4 名注册用户`
- `3 名付费用户`
- `累计生成 28 条报告`

这些数字证明的不是“规模已经很大”，而是这个项目已经跨过“只会做 demo”的阶段，开始验证：

- 产品方向是否有人愿意使用
- 交付链路是否真的能跑通
- 用户是否愿意为这类研究工作流付费

它证明了产品方向、完整交付能力和初步付费可行性。
